5 # Running MID workflow
6 
7 The MID clustering takes as input the "digits" produced by the simulation and returns the recontructed tracks and the associated MC labels.
8 
9 ## Getting the digits
10 If you do not have the digits, you can obtain a sample with:
11 ```bash
12 o2-sim -g fwmugen -m MID -n 100
13 o2-sim-digitizer-workflow
14 ```
15 
16 ## Execution
17 To run the reconstruction workflow, run:
18 ```bash
19 o2-mid-reco-workflow
20 ```
21 
22 Options:
23 
24 `--disable-mc`
25 : do not propagate the MC info (faster: this is what happens with real data)
26 
27 ### CAVEATs
28 
29 * The termination of the workflow is currently tricky. So this workflow does not automatically quit when the processing is done. The program should be therefore either quit manually or one needs to specify a number of events to analyse with option: `--nevents <number_of_events>`
30 * Also, the reconstruction part is way faster than the label propagation part. So it can happen that the MC labels are dropped before being processed. This issue should be solved at the level of DPL. In the meanwhile a temporary solution consists in changing the calue of DEFAULT_PIPELINE_LENGTH in the [DataRelayer](../../../../Framework/Core/src/DataRelayer.cxx). A value of 64 was found to be enough.
